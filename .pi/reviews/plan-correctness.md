## Critical Finding: P1-3 `ctx.model?.id` May Be Wrong Model

**Severity:** 🔴 HIGH — could silently break the weak-model detection

**File:** Plan section P1-3, `refineMessageIfGeneric`

**Problem:** The plan uses `ctx.model?.id` to detect weak models:

```typescript
if (isCheapModel(ctx.model?.id ?? "")) {
  return userScore > genScore ? userCandidate : generatedMessage;
}
```

But the model that **actually generated** the commit message may differ from `ctx.model`. The resolution chain is:

1. `aiComplete(ctx, ...)` internally calls `resolveModel(ctx)` (see `src/core/ai.ts` line 33)
2. `resolveModel` first checks the configured `analysis_model` setting; only falls back to `ctx.model` (see `src/core/resolve-model.ts` lines 22–53)
3. `aiComplete` returns `{ text, model }` with the **actual** model used, but `generateAutoCommitMessage` discards `result.model`

So if the user has configured `analysis_model = "deepseek/deepseek-v4-pro"` but their session model is `gpt-5.4-mini`:
- The commit message is generated by `deepseek-v4-pro` (strong)
- `isCheapModel(ctx.model?.id)` checks `gpt-5.4-mini` → returns `true` → skips AI comparison
- Wrong! Should be using the deepseek model for comparison since it generated the message

**Fix:** Use the actual model from `aiComplete`'s return value. Either:

**Option A:** Pass the model ID from `generateAutoCommitMessage`:
```typescript
// In generateAutoCommitMessage:
const result = await aiComplete(ctx, { ... });
// ...
return await refineMessageIfGeneric(
  ctx,
  commitMessage,
  messages,
  changedFiles,
  lang,
  result?.model?.id,  // NEW parameter
);
```

**Option B:** Call `resolveModel(ctx)` directly in `refineMessageIfGeneric` (import from `resolve-model.js`):
```typescript
import { resolveModel } from "./resolve-model.js";
// ...
const model = resolveModel(ctx);
if (model && isCheapModel(model.id)) { ... }
```

Option B is simpler and doesn't require changing the function signature. It also resolves the same way `aiComplete` does.

## Verified: P0-5 Newest-First Budget — Correct

**Both `reverse()` calls correctly removed.** Traced through the actual source at `src/core/auto-commit-message.ts` lines 317–339:

**Current flow (broken):**
```
userMessages = [newest, ..., oldest]     ← collectMessagesByRole returns newest-first
userMessages.reverse() → [oldest, ..., newest]  ← iterate oldest-first
userLines = [oldest, ..., maybe_newer]   ← budget exhausted on old messages
userLines.reverse() → [maybe_newer, ..., oldest]
userStr = userLines.reverse().join()     ← display newest-ish first
```
→ Newest messages (most relevant) may get ZERO budget.

**After fix (correct):**
```
userMessages = [newest, ..., oldest]     ← no reverse, iterate newest-first
userLines = [newest_trimmed, ..., oldest_trimmed]
userStr = userLines.join()               ← display newest first (no reverse needed)
```
→ Newest messages get budget first, most recent context preserved. ✅

The plan says "Same pattern for assistant section" — this is correct for lines 332–339. Both `reverse()` on the iteration and `reverse()` on the join must be removed for both sections.

## Missing Issues — Should Be Addressed

### 1. Type Hints (Prompt-Engineering #4) — Should be P1

**Severity:** 🟡 HIGH — small models benefit significantly from explicit type guidance

The plan omits file-path-based type hints entirely. `inferTypeFromFiles` already exists in `src/core/commit-message.ts` (lines 54–74) and is trivially reusable. The review report suggests adding type hints to the prompt as a prefix:

```typescript
// In buildPrompt(), after building sections:
const typeHint = buildTypeHintForMessage(changedFiles);
// Prepend to user prompt
```

This is ~15 lines of code for meaningful improvement for small models. The plan should at minimum add this as P1-5 or explain why it's excluded.

**Recommendation:** Add as P1-5.

### 2. Sanitize First-Line Defense (Prompt-Engineering #7) — Note as P2 Follow-up

The plan relies on `cleanCommitOutput` (P0-1) to handle multi-line output. If `cleanCommitOutput`'s Layer 4 fallback returns a multi-line string (unlikely but possible), `sanitizeCommitMessage` in `commit-message.ts` would process it incorrectly due to `$` not matching with `m` flag.

This is defense-in-depth. Low risk given `cleanCommitOutput`'s robustness, but worth noting as a future safeguard.

**Recommendation:** Add as P2-4.

### 3. Budget Truncation #3 (Semantic Awareness) — Acceptable Deferral

With the assistant budget increasing from 600 to 2500 in P0-4, truncation will be triggered much less frequently for the most valuable section. Deferring this is reasonable. If it becomes an issue post-P0, it can be prioritized.
